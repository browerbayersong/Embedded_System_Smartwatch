# 🛠️ 组员 B 固件与 RTOS 开发指南

## 一、 STM32CubeMX 关键接口配置

### 1. I2C 接口配置（连接 MPU6050 与 OLED）

从组员 A 提供的原理图（I2C电路、OLED电路）来看，**MPU6050 和 OLED 共用了一组 I2C 总线，并且引脚上拉电阻（$4.7\text{k}\Omega$）接在 PB10 (SCL) 和 PB11 (SDA) 上**。

* **引脚映射（Remap）**：在 CubeMX 中开启 I2C1 后，标准默认引脚通常是 PB6/PB7。**必须手动在管脚图上将 I2C1 重新映射到 PB10 (SCL) 和 PB11 (SDA)**，否则无法与硬件连通。
* **速度设置（Speed）**：速率建议设为 **Fast Mode (400kHz)**。由于 OLED 屏幕刷新和陀螺仪高频数据采集共用一条总线，标准 100kHz 可能会导致界面刷新有肉眼可见的卡顿。
* **从机地址（Slave Address）确认**：
* **MPU6050**：原理图中 AD0 引脚接 GND，其 7 位 I2C 地址为 `0x68`。在调用 STM32 HAL 库函数时，需要将其左移一位，即传入 `(0x68 << 1)`（即 `0xD0`）。
* **OLED 屏幕**：根据电路图上的 `HS096L03W2C03` 4针接口，市面同类单色 OLED 的 7 位 I2C 地址通常为 `0x3C`，在 HAL 库中对应 `(0x3C << 1)`（即 `0x78`）。



### 2. 系统与时钟配置（SYS & RCC） [⚠️ 核心排坑点]

* **时钟源（RCC）**：根据最小系统板电路，选择外部高速晶振（HSE），并在 Clock Configuration 中将系统主频（SYSCLK）配置为最高的 **72MHz**，以保证 RTOS 多任务运行时的算力。
* **系统心跳定时器（SYS -> Timebase Source）**：
* **致命冲突**：FreeRTOS 默认会占用 `SysTick` 滴答定时器作为自己的系统心跳（Ticks）。如果 STM32 HAL 库的延迟函数（如 `HAL_Delay`）也使用 `SysTick`，会导致严重的优先级冲突，引发系统死锁。
* **解决方案**：在 CubeMX 的 `SYS` 配置中，**必须将 `Timebase Source` 从默认的 `SysTick` 更改为任意一个通用定时器（例如 TIM1 或 TIM4）**。



### 3. FreeRTOS 基础配置

* **API 版本**：在 Middleware 中开启 FreeRTOS，建议选择 `CMSIS_V1`（标准且稳定，更便于团队协作调试）。
* **内存管理（Memory Management）**：根据项目基本原理要求，**必须选择 `heap_4` 策略**。它会按物理地址排序管理空闲块，能有效合并相邻空闲内存，减少系统运行时的内存碎片。
* **初始任务（Tasks）划分**：在 Tasks and Queues 页面中，先配置好以下三个基础任务：
1. `task_time`（时间管理，优先级：Medium）
2. `task_sensor`（传感器采集，优先级：Low/Medium）
3. `task_ui`（UI 显示刷新，优先级：Low，因为 OLED 刷新较占总线时间）

## 二、 固件开发与代码编写注意事项

### 1. I2C 总线多任务冲突（Mutex 互斥锁）

> 🚨 **核心陷阱**：`task_sensor` 需要高频从 I2C 读取陀螺仪数据，而 `task_ui` 需要通过 I2C 往 OLED 写数据。如果两个 RTOS 任务在同一时刻调用 HAL 库的 I2C 读写函数，会导致 I2C 硬件外设直接死锁或数据严重错乱。

* **解决方案**：必须在 FreeRTOS 中创建一个 **互斥量（Mutex）**。无论是初始化、读取 MPU6050 还是刷新 OLED，**所有对 I2C 总线的操作前后，都必须成对使用互斥锁进行保护**：
```c
if (osMutexWait(I2cMutexHandle, osWaitForever) == osOK) {
    HAL_I2C_Master_Transmit(&hi2c1, ...);
    osMutexRelease(I2cMutexHandle);
}

```


* **超时时间（Timeout）**：调用 `HAL_I2C_Master_Transmit` 或 `HAL_I2C_Master_Receive` 时，最后的 `Timeout` 参数**绝对不能**写 `HAL_MAX_DELAY`。请设定一个合理的毫秒数（如 `10` 或 `50`），防止因硬件拔插或接触不良导致整个系统卡死。

### 2. 软件时间基准与传感器采集优化

* **软件计时（task_time）**：项目要求不使用额外 RTC 芯片实现软件计时。
* 可以在 `task_time` 中利用 FreeRTOS 提供的精准绝对延时函数 `vTaskDelayUntil()`，设定为 `1000ms`（1秒）触发一次，并在任务中对自定义的时间结构体（时、分、秒）进行累加。


* **传感器数据传递**：
* MPU6050 的数据采集不需要高频刷新（通常 20Hz~50Hz 即可满足组员 C 的计步逻辑）。
* 采集到的原始数据**不要直接用全局变量共享**。建议利用 FreeRTOS 的 **消息队列（Queue）** 将数据发送给组员 C 的 `pedometer.c` 算法模块，保证线程安全。

## 三、 VSCode 环境与 Makefile 联调注意

由于团队采用 **VSCode + GCC + Makefile** 的本地工具链，你在生成和管理代码时需要注意：

1. **Project Manager 设置**：在 CubeMX 导出工程时，`Toolchain/IDE` 必须选择 **Makefile**。
2. **代码沙盒保护**：每次修改 CubeMX 配置并重新生成代码时，确保你自己编写的初始化或逻辑代码严格写在 `/* USER CODE BEGIN */` 和 `/* USER CODE END */` 之间。你自己新建的驱动文件（如 `mpu6050.c`, `oled.c`）千万不要放在 `Src` 根目录下被误刷，建议建立独立的文件夹或做好备份。
3. **手动修改 Makefile**：当你新建了 `i2c_drv.c`、`mpu6050.c`、`oled.c` 等驱动文件后，VSCode 的编译器并不知道它们的存在。**你必须手动打开项目根目录下的 `Makefile**`，在 `C_SOURCES` 中添加这些 `.c` 文件的路径，在 `C_INCLUDES` 中添加对应的 `.h` 文件夹路径，否则编译时会爆出 `Undefined reference` 错误。

## 💡 组员 B 今天的立刻行动指南

1. 打开 CubeMX，芯片选型 `STM32F103C8T6`。
2. 配置 RCC 外部晶振，在 Clock 树中将主频拉满到 72MHz。
3. 将 SYS Timebase Source 修改为 `TIM1`（或其它非 SysTick 定时器）。
4. 开启 I2C1 并引脚重映射到 `PB10/PB11`。
5. 启用 FreeRTOS（CMSIS_V1），创建好三个初始任务。
6. 导出 **Makefile** 工程，在 VSCode 中直接运行 `make` 编译，确保空工程无误后，将 `project.ioc` 提交到你的 `feature/firmware-freertos` 分支！
